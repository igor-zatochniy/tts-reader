# Безпека

## Модель загроз

Проєкт працює як локальний сервіс і має доступ до файлової системи користувача та до локального TTS engine. Тому основні ризики:

- доступ з браузера до локального API;
- запуск дій без явного дозволу користувача;
- читання або зміна не тих файлів;
- підміна progress-файлів;
- зависання shutdown через довгі з’єднання.

## Локальна модель загроз

Окремо варто враховувати, що цей сервіс живе на машині користувача, а не в ізольованому хмарному середовищі. Це означає:

- браузер може виконати запити до `127.0.0.1`, якщо сторінка не захищена належним чином;
- локальний процес може читати та змінювати файли книги й progress;
- помилки з файлової системи не можна показувати клієнту в сирому вигляді;
- токен доступу не повинен залишатися в адресному рядку або в історії браузера;
- перевірка секретів має охоплювати всю Git-історію, а не лише поточний snapshot.

## Що вже захищено

### Loopback only

Сервер приймає лише loopback-адресу типу `127.0.0.1:8080`.

Це зменшує ризик випадкового відкриття сервісу в локальній мережі.

### Host та Origin перевірки

- `Host` має бути loopback.
- `Origin` має бути дозволеним і також loopback.

Це зменшує ризик небажаних запитів із браузера.

### API token

- Для state-changing запитів використовується token.
- Токен передається через `X-TTS-Token` або query string для браузерної панелі.
- Після завантаження сторінки токен прибирається з адресного рядка через `history.replaceState(...)`.

### Безпечні HTTP-помилки

- Клієнт отримує лише безпечні повідомлення та стабільні коди помилок.
- Внутрішні деталі на кшталт raw `err.Error()` не повертаються напряму.
- Повні причини збою логуються на стороні сервера.

### JSON validation

- Body обмежений по розміру.
- `Content-Type` має бути `application/json`.
- JSON декодується з `DisallowUnknownFields()`.

### File identity checks

- При старті читання і при `SetPosition` перевіряється актуальна ідентичність книги.
- Якщо файл змінився, операція відхиляється.

### Public JSON без filesystem paths

- У зовнішніх відповідях повертаються лише безпечні поля.
- Абсолютні шляхи книги не віддаються через API.

## Що робить сервіс безпечнішим

### Stop і shutdown

- `Stop` чекає завершення поточної session goroutine.
- `server.Shutdown(...)` запускається після скасування `BaseContext`.
- Це зменшує шанс залишити довгий SSE-запит або стару playback-сесію живими.

### Progress file

- Progress записується локально в JSON.
- Використовується атомарна заміна файла.
- Тимчасовий файл не має бути кінцевим джерелом істини.

### Windows TTS

- Текст передається через `stdin`, а не через довгі environment variables.
- Це зменшує шанс зламу запуску через обмеження на розмір змінних середовища.

### Full history secret scan

- У CI використовується Gitleaks.
- Перевірка запускається з `fetch-depth: 0`, щоб сканувати всю Git-історію.
- Додатково виконується явна команда `gitleaks detect --source . --verbose`.
- Це допомагає ловити не лише поточні секрети, а й ті, що могли залишитися в старих комітах.

## Обмеження

Це локальний desktop-сервіс, не публічний multi-user API.

Тому тут немає:

- окремої автентифікації користувачів;
- ролей;
- мережевої авторизації;
- захисту від довіреного локального користувача з доступом до процесу.

## Практичне правило

Усе, що змінює стан або торкається файлової системи, має проходити через:

1. loopback;
2. token;
3. JSON validation;
4. перевірку book identity;
5. локальне атомарне збереження.
